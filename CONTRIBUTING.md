# Contributing to grtblog-v2

Thank you for your interest in contributing to **grtblog-v2**. This document covers the development setup, architecture overview, key conventions, and contribution workflow.

---

## Technology Stack

This project is organized as a monorepo with three main modules:

| Module | Path | Stack | Description |
|--------|------|-------|-------------|
| **Server** | `/server` | Go 1.24+, Fiber, GORM, PostgreSQL | Backend API with DDD architecture; handles data management and drives static page generation. |
| **Admin** | `/admin` | Vue 3, Naive UI, Vite | Content management dashboard with Markdown editing. |
| **Web** | `/web` | SvelteKit, Tailwind CSS | Reader-facing frontend. Runs SSR in production; static snapshots are generated by the backend. |

---

## Development Setup

### Prerequisites

- Go 1.24+
- Node.js 20+
- pnpm 9+
- PostgreSQL

### 1. Start the Backend (Server)

The backend serves as the data source and runs on port **:8080**.

```bash
cd server
cp .env.example .env   # Configure database credentials
make migrate-up         # Run database migrations
make run
```

### 2. Start the Admin Dashboard

The admin dashboard runs on port **:5799**.

```bash
cd admin
pnpm install && pnpm dev
```

### 3. Start the Web Frontend

- **Development mode** (port **:5173**): hot-reload enabled.
- **Production SSR** (port **:3000**): used for ISR static generation testing.

```bash
cd web
pnpm install && pnpm dev
```

## ISR (Incremental Static Regeneration) Testing

The project implements a custom ISR mechanism: the Go backend scrapes the SSR server (port :3000) and persists the rendered HTML as static files.

If you modify **SvelteKit routing logic** or **backend generation code**, you must verify the ISR pipeline.

### Running the ISR Preview

The ISR preview script requires an admin token for API authentication. Create one via the Admin dashboard (**Settings > Admin Tokens**), then run:

```bash
# Ensure the backend is running on :8080
export PREVIEW_ISR_TOKEN="gt_your_admin_token_here"
make preview-isr
```

The script performs the following steps:

1. Cleans the HTML storage directory.
2. Builds the Web frontend.
3. Copies static assets to `server/storage/html/`.
4. Starts the SSR server on port :3000.
5. Calls the backend API to generate static HTML snapshots.
6. Stops the SSR server and launches a static file server on port **:5555**.

> The ISR pipeline is working correctly if all pages render properly on :5555.

## Development Guidelines

### Backend: Synchronous vs. Asynchronous Generation

- **API handler (`RefreshPostsHTML`)**: Must be **synchronous**. The caller blocks until generation completes and receives a 200 response, allowing the script to know when to shut down the SSR server.
- **Event subscriber (`ArticleUpdated`)**: Must be **asynchronous**. Triggered via EventBus on article save; must not block the save request.

### Frontend: Trailing Slashes

SvelteKit is configured with `trailingSlash: 'always'`. All generated URLs must end with `/` (e.g., `/posts/hello/`) to match the `posts/hello/index.html` file structure. Without this, static deployments will return 404 on page refresh.

### Shared Markdown Components

Custom Markdown components (albums, cards, etc.) are defined in `shared/markdown/components.ts`. This code is shared between Admin and Web. When adding or modifying components, ensure both modules are updated accordingly.

## Commit Conventions

This project follows [Conventional Commits](https://www.conventionalcommits.org/):

| Prefix | Usage |
|--------|-------|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation changes |
| `refactor` | Code restructuring without behavior change |
| `chore` | Build, CI, dependency updates |

Example: `feat(server): add word count statistics for articles`

## Release Process

The project uses a **unified version number** across all modules.

- **Format**: `vMAJOR.MINOR.PATCH` (e.g., `v2.1.0`)
  - `MAJOR`: Breaking changes (incompatible API or DB migration changes)
  - `MINOR`: Backward-compatible new features
  - `PATCH`: Backward-compatible fixes and minor improvements
- **Pre-release**: Use SemVer pre-release tags (e.g., `v2.1.0-beta.1`, `v2.1.0-rc.1`)
- **Intermediate builds**: Reference by commit hash (e.g., `8f3c1a2b9d4e`)

### Creating a Release

```bash
# Create release tag locally
./scripts/release.sh v2.1.0

# Create and push release tag
./scripts/release.sh v2.1.0 --push
```

The release script will:

1. Validate the version format.
2. Generate a release notes draft at `docs/releases/<version>.md`.
3. Create an annotated Git tag.

### Image Tags

Docker image tags must match the release version:

- `grtblog-server:v2.1.0`
- `grtblog-renderer:v2.1.0`

The `APP_VERSION` variable in `deploy/.env` controls the deployed image version.
